# Reflection

The biggest technical challenge was building a downloader that was both fast and easy to control and audit. At first, multithreading seems like just starting several downloads at once. In reality, I had to handle three separate issues: limiting how many downloads run at the same time, making sure I followed the request-per-minute rule, and keeping shared output safe. I solved these with a thread pool, a semaphore, a sliding-window rate limiter, and a lock for the log file. Splitting up these tasks made the program easier to understand. I also learned that concurrency is not just about speed—it introduces new problems that don't exist in single-threaded code.

Using multiple threads made the API download faster, and adding synchronization made it safer. Threads are helpful because API calls spend a lot of time waiting for responses, so another request can move forward while one waits. I used a benchmark to show this instead of just assuming threading would help. But threads alone are not enough to keep things safe. Without a semaphore, the program could send too many requests at once. Without a rate limiter, it might break the API rules. Without a log lock, several threads could write to the log at the same time and make a mess. In the end, I treated speed and safety as separate but equally important goals.

One useful lesson was that a benchmark must measure the correct section. Including CSV writing or analytics would make it unclear whether the difference came from downloading or another task. The coursework specifically asks for the download step only, so the timer surrounds the serial or threaded API work and stops before the final dataset is written. I would repeat each benchmark several times in a production study because a single network measurement can be affected by temporary conditions.

Pandas made inspection and cleaning much easier. Its data-frame view allowed me to print dimensions, preview records, inspect data types and count missing values with concise operations. Safe numeric and date conversion was especially valuable. Instead of writing many manual try/except blocks, invalid values could be converted to missing values and handled consistently. String methods also made symbol standardisation straightforward. Duplicate detection, filtering and engineered columns were readable because each transformation closely matched the business rule.

The data-quality report was important because simply producing a smaller clean file would not explain what happened. Recording counts before and after cleaning showed how many duplicates, invalid numbers, dates and negative volumes were found. This creates an audit trail and helps detect an overly aggressive cleaning rule. I also learned that dropping invalid rows is a judgement, not an automatic solution. For market candles, inventing prices or timestamps could distort later analytics, so removing records with essential unrecoverable fields was safer than filling them with averages.

The balanced 50-record pandas sample demonstrated another analytical issue: a sample should match its purpose. Randomly taking 50 rows from the entire file might under-represent some symbols. Taking five per symbol guarantees coverage and makes the comparison more useful. At the same time, I kept this sample separate from the final conclusions because it cannot represent the full dataset reliably.

Spark made full-dataset aggregation and repeatable SQL analysis easier. Registering the cleaned data as a temporary view meant I could express business questions directly: which symbol was most volatile, which had the greatest trading activity, when were the busiest hours, and how many candles moved up or down? Spark also made it simple to group, sort and rank results across all symbols. The current dataset is small enough for pandas, but the point of Spark is that the same approach can scale to much larger histories or many more markets.

I found the separation between pandas and Spark useful. Pandas is excellent for detailed cleaning and quick local inspection. Spark is more appropriate for scalable aggregation and distributed processing. Using both showed that tool choice should depend on the stage rather than using one technology for everything.

With more time, I would improve the project in five ways. First, I would add automated tests for field mapping, rate-limit behaviour and cleaning rules. Second, I would retry temporary API failures with exponential backoff while still failing clearly after a maximum number of attempts. Third, I would store the cleaned data in Parquet to preserve types and improve Spark read performance. Fourth, I would execute the Spark notebook automatically in continuous integration and archive its outputs. Finally, I would add charts for volatility and activity, while keeping the CSV summary as the auditable source.

The project strengthened my understanding of complete data pipelines. Reliable analytics depends on every earlier stage: controlled collection, explicit validation, traceable cleaning and clear interpretation. The main achievement was not a single query but a workflow in which each team can understand and trust the handoff it receives.
